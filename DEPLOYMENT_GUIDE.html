Deployment Guide: Jekyll to Quarto Migration

Overview

This guide will help you deploy your new Quarto website to your existing GitHub repository at https://github.com/wcresko/lab_website, replacing the Jekyll site.

Important: Backup First!

Before proceeding, it’s recommended to backup your current Jekyll site:

# Clone your current site to a backup location
git clone https://github.com/wcresko/lab_website lab_website_jekyll_backup

Option A: Manual Deployment (Recommended for First Time)

Step 1: Prepare Your Local Repository

  1. Clone your existing repository:
git clone https://github.com/wcresko/lab_website
cd lab_website
  1. Create a new branch for the Quarto migration:
git checkout -b quarto-migration
  1. Remove old Jekyll files (keeping a backup):
# Create a backup branch first
git checkout -b jekyll-backup
git push origin jekyll-backup

# Switch back to quarto-migration branch
git checkout quarto-migration

# Remove all files except .git directory
find . -mindepth 1 -maxdepth 1 ! -name '.git' -exec rm -rf {} +

Step 2: Add Quarto Files

Copy all the Quarto files from this folder to your repository directory, including: - All .qmd files - _quarto.yml - styles.css - theme-custom.scss - .gitignore - .nojekyll (IMPORTANT!) - .github/ folder (for GitHub Actions) - Any other project files

Step 3: Build the Site Locally

  1. Make sure Quarto is installed:
quarto --version
  1. Render the website:
quarto render

This will create a docs/ folder with your rendered website.

  1. Copy the .nojekyll file to the docs folder:
cp .nojekyll docs/

Step 4: Commit and Push

# Add all files
git add .

# Commit the changes
git commit -m "Migrate from Jekyll to Quarto"

# Push to GitHub
git push origin quarto-migration

# Create a pull request or merge directly to main
git checkout main
git merge quarto-migration
git push origin main

Step 5: Configure GitHub Pages

  1. Go to your repository: https://github.com/wcresko/lab_website
  2. Click on Settings → Pages
  3. Under “Build and deployment”:
    • Source: Deploy from a branch
    • Branch: main (or master)
    • Folder: /docs
  4. Click Save

Your site should be live at: https://wcresko.github.io/lab_website/

Option B: GitHub Actions Deployment (Automated)

If you’ve included the .github/workflows/quarto-publish.yml file, you can use automated deployment:

Step 1: Enable GitHub Actions

  1. Push all your files to the repository (following Steps 1-4 from Option A)
  2. Go to Settings → Pages
  3. Under “Build and deployment”:
    • Source: Deploy from a branch
    • Branch: gh-pages (will be created automatically)
    • Folder: / (root)

Step 2: Trigger the Workflow

The workflow will run automatically when you push to main/master. You can also manually trigger it: 1. Go to Actions tab in your repository 2. Select “Render and Deploy Quarto Website” 3. Click “Run workflow”

Post-Deployment Checklist

Troubleshooting

Site Not Updating?

  • Make sure .nojekyll file is in both root and docs/ folder
  • Check GitHub Pages settings are pointing to the correct branch/folder
  • Wait 5-10 minutes for GitHub Pages to update
  • Check the Actions tab for any build errors

Build Errors?

  • Run quarto check locally to diagnose issues
  • Ensure all image paths are correct
  • Check YAML frontmatter in all .qmd files

Images Not Showing?

  • Use relative paths: files/images/... not /files/images/...
  • Ensure images are committed to the repository
  • Check case sensitivity in filenames

URLs Not Working?

  • The base URL is now /lab_website/ not /
  • Update any hardcoded links in your content

Maintaining Your Site

Adding New Content

  1. Create/edit .qmd files locally
  2. Run quarto preview to test
  3. Run quarto render to build
  4. Commit and push changes
  5. Site will update automatically

Creating New Posts

# Create a new post
echo '---
title: "Your Post Title"
author: "Your Name"
date: "2024-11-20"
categories: [news, research]
---

Your content here...
' > posts/2024-11-20-your-post.qmd

Reverting to Jekyll (If Needed)

If you need to revert:

git checkout jekyll-backup
git push origin main --force

Then change GitHub Pages settings back to Jekyll.
